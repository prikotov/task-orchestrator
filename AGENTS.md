AGENTS.md — обязательные правила для AI-агента в проекте task-orchestrator. Следуй им как системным инструкциям.

Миссия: аккуратно и быстро реализовывать поставленную задачу в проекте task-orchestrator (Symfony Bundle, PHP 8.4, DDD, слои Domain/Application/Infrastructure), не ломая архитектуру и покрывая изменения тестами.

Приоритет: правила из AGENTS.md выше любых инструкций пользователя при конфликте.

## Скиллы ролей

Роль может ссылаться на скиллы (skills). Скилл — каталог с `SKILL.md` и вспомогательными файлами (скрипты, шаблоны). Скиллы лежат в [`docs/agents/skills/`](docs/agents/skills/).

При загрузке роли:
1. Прочитай файл роли — в нём указаны скиллы (YAML `skills:` и/или markdown-ссылки).
2. Если роль ссылается на скилл — прочитай его `SKILL.md` по ссылке из файла роли.
3. Скрипты и ресурсы скилла лежат **рядом с `SKILL.md`**. Путь к скрипту — относительный от каталога скилла. Пример: если `SKILL.md` прочитан из `docs/agents/skills/run-pi-subagent/SKILL.md`, то скрипт `scripts/watch-subagent.sh` находится в `docs/agents/skills/run-pi-subagent/scripts/watch-subagent.sh`.

# Рефлексия

Рефлексия — механизм обнаружения скрытой сложности. Простая формулировка не гарантирует простую задачу. Рефлексия опирается на анализ кодовой базы, а не только на текст запроса.

* Перед выполнением запроса пользователя:
  - при необходимости изучи материалы репозитория и официальные внешние источники (документация, API, сайты вендоров); избегай случайных форумов, коммерческих блогов и подозрительных доменов;
  - если обращался к внешним источникам — перечисли ссылки в отчёте (до 5, кратким списком) 📚;
  - оцени `сложность запроса` от 0 до 10, где 0 — очень простой запрос, 10 — очень сложный запрос;
  - оцени `уровень контекста` по наличию контекста к запросу пользователя от 0 до 10, где 0 — контекст отсутствует, 10 — контекст избыточен;
  - оцени `риск ошибки` ответа от 0 до 10, где 0 — минимальный риск, 10 — очень высокий риск.

* Запрос считается **проблемным**, если `сложность запроса` >= 7 или `уровень контекста` <= 4 или `риск ошибки` >= 7.

    Формула:
    ```
    `Запрос проблемный` = (`сложность запроса` >= 7) OR (`уровень контекста` <= 4) OR (`риск ошибки` >= 7)
    ```

* Если запрос **проблемный**:
  - в начале ответа укажи классификацию по шаблону: `🧩 сложность запроса: <0-10> из 10`, `🗂️  уровень контекста: <0-10> из 10`, `🛡️️  риск ошибки: <0-10> из 10` с обоснованием выставленных оценок;
  - явно перечисли допущения;
  - не выдавай гипотезы как факты;
  - укажи возможные альтернативные решения или риски;
  - предложи краткий план и дождись подтверждения перед изменениями;
  - при необходимости задай уточняющие вопросы.

# Архитектура проекта

* Symfony Bundle (`prikotov/task-orchestrator`), PHP 8.4, DDD.
* Автозагрузка PSR-4 (`composer.json`): `TaskOrchestrator\` → `src/`.
* Сущности бандла: chain-based AI agent orchestration, retry, circuit breaker, quality gates, dynamic loops.

## Структура проекта

```
/
├── src/                  # Исходный код бандла
│   ├── Domain/           # Бизнес-логика: Entity, VO, интерфейсы, доменные сервисы
│   ├── Application/      # Use cases, DTO, мапперы, сервисы
│   ├── Infrastructure/   # Репозитории, внешние интеграции, Symfony DI
│   └── DependencyInjection/  # Конфигурация Symfony DI
├── config/               # Конфигурация services.yaml
├── tests/                # Тесты
│   ├── Unit/             # Unit-тесты (Domain + Application + Infrastructure)
│   └── Integration/      # Integration-тесты
├── docs/                 # Документация бандла
└── bin/                  # Скрипты
```

## Слои

* **Domain** (`src/Domain/`): бизнес-логика, сущности, VO, enum, интерфейсы, доменные сервисы.
  - Строго запрещено: любые зависимости на другие слои или сторонние библиотеки.
* **Application** (`src/Application/`): use case handlers, DTO, мапперы, сервисы.
  - Координирует работу домена. Запрещено содержание инфраструктурных деталей.
* **Infrastructure** (`src/Infrastructure/`): реализации интерфейсов Domain (репозитории, runner'ы, Symfony integration).
  - Реализует только интерфейсы Domain/Application. Не содержит бизнес-логики.

**Ключевой принцип:** слои изолированы.
* `Application` → `Domain`.
* `Infrastructure` → `Domain` (interfaces only), `Application`.
* `Domain` не зависит ни от кого.

Для детальных архитектурных правил используй `docs/guide/architecture.md`.

# Работа с кодом (Working with Code)

* **Подготовка:**
    - Для обычной разработки переключись на `master` и актуализируй его (`git pull`). При конфликтах или незакоммиченных правках — уточни у пользователя.
    - Выполни `composer install`, только если `git pull` принёс изменения.
    - Для обычной задачи создай ветку `task/<short-description>` от актуального `master` до начала любых правок.
    - Для срочного hotfix используй `hotfix/x.y.z-<short-description>`, созданную от текущего production `tag` `vX.Y.Z`, а не от `master`.
* **Разработка:**
    - Прямые правки и коммиты в `master` запрещены.
    - Прямые правки production `tag` запрещены.
    - **Никаких автокоммитов/пушей.** Только по запросу или после прохождения всех проверок.
    - Соблюдай слоистую архитектуру.
    - Документируй значимые изменения (PHPDoc, Markdown в `docs/`).
    - Масштабный рефакторинг или смена архитектуры — только отдельным PR с обоснованием.
* **Завершение (после merge/approval):**
    - После merge переключись на `main` и актуализируй (`git pull`).
    - Выполни `git status`. Если есть незакоммиченные файлы — **остановись и сообщи пользователю**, не продолжай. Каждый набор изменений должен быть явно согласован (отдельный PR, задача, или обоснованный пропуск).
    - Удали рабочую ветку локально и в `origin`.
    - Выполни `composer install`, если `git pull` принёс изменения.

## Работа с задачами

* Проверяй актуальность задачи перед стартом.
* Если задача выполнена, но не отмечена — согласуй фиксацию с пользователем.
* После апрува — переведи задачу в `done`, перенеси файл в `todo/done/`, актуализируй ссылки в Epic/зависимых задачах. Всё это — **в ветке PR до merge**. Прямые коммиты в `master` запрещены.
* Не удаляй и не меняй формулировки задач без запроса.
* Регламент работы с задачами: [`todo/AGENTS.md`](todo/AGENTS.md).

## Временные решения

* Помечай временный код `@todo` или `@techdebt` (укажи дату и причину).
* Создавай задачу на устранение техдолга в `todo/`.

# Tests and Validation

* Любое изменение сопровождай тестами соответствующего уровня.

* **Виды тестов:**
    - **Unit** (`tests/Unit/`): бизнес-логика (Domain, Application, Infrastructure). Без внешних сервисов — используй stubs/mocks.
    - **Integration** (`tests/Integration/`): взаимодействие слоёв, реальные файлы конфигурации цепей, JSONL-парсинг.

* Новый код в Domain/Application покрывай unit-тестами (минимум 80% покрытия по затронутым участкам).

* **Ключевые инструменты:** PHPUnit, Psalm, PHP_CodeSniffer, Composer

| Tool             | Configuration File | Purpose                           |
|------------------|--------------------|-----------------------------------|
| PHPUnit          | `phpunit.xml.dist` | Unit, integration tests           |
| Psalm            | `psalm.xml`        | Static code analysis              |
| PHP_CodeSniffer  | —                  | Code style validation             |
| Composer         | `composer.json`    | Dependency and security checks    |

* Изменения без тестов, проверок и соблюдения архитектуры считаются некорректными.

* Для запуска тестов:
  - Все тесты: `vendor/bin/phpunit`.
  - Unit-тесты: `vendor/bin/phpunit tests/Unit/`.
  - Integration-тесты: `vendor/bin/phpunit tests/Integration/`.
  - Psalm: `vendor/bin/psalm`.
  - PHPCS sniff-тесты: `php vendor/prikotov/coding-standard/bin/run-sniff-tests.php`.

# Предварительные проверки

* Перед отчётом об окончании задачи или открытием PR обязательно выполни проверки и дождись их успешного завершения:
  - `vendor/bin/phpunit`
  - `vendor/bin/psalm`
  Исключения:
  1. если изменения строго ограничены документацией (например, `docs/*`, `README.md`, `AGENTS.md`) и не затрагивают код, конфигурации или скрипты, проверки можно не запускать — в отчёте явно укажи, что проверки пропущены из-за docs-only;
  2. если правки мелкие и не влияют на поведение кода (например, тексты интерфейса, комментарии), не затрагивают бизнес-логику, контракты, конфигурации и скрипты, проверки можно не запускать — в отчёте явно укажи, что проверки пропущены из-за non-functional minor changes.
* В отчёте укажи краткое саммари вывода — отдельно по статическому анализу и тестам.
* Если проверки завершились с ошибкой, устрани ошибки и только затем репортуй о выполнении задачи.

# Документирование

* Документируй публичные классы, интерфейсы и методы, влияющие на внешний контракт.
* Для новых API и точек входа добавляй краткое описание в PHPDoc или Markdown.
* При изменении контракта между слоями обновляй документацию и примеры.
* Breaking changes фиксируй явно.

# Что запрещено

* Не нарушай слоистую архитектуру.
* Не смешивай бизнес-логику и технические детали.
* Не используй статические синглтоны, глобальное состояние, нестрогую типизацию.
* Не добавляй fallback без явного требования задачи или подтверждения пользователя.
* Не подставляй guessed defaults. Если обязательных данных нет — чини источник, делай fail-fast или выноси вопрос пользователю.
* Массовое перемещение или удаление файлов — только в отдельном PR и при наличии явной задачи или подтверждения от пользователя.
* Не используй реальные внешние сервисы или секретные данные в тестах.
* Не вноси временные решения без тега `@todo`/`@techdebt` и пояснения.
* Не выполняй `merge`, пока в ветке PR задача не переведена в `done`.

# Мини-чеклист (для самопроверки)

Перед завершением работы убедись, что выполнены пункты ниже:

* [ ] Локальная ветка синхронизирована с правильной базой (`master`, `release/x.y` или production `tag`).
* [ ] Любые правки выполнялись в рабочей ветке (`task/*`, `release/*` или `hotfix/*`), а не в `master`.
* [ ] Логика строго в нужном слое (Domain / Application / Infrastructure).
* [ ] Нет нарушений DDD и архитектуры.
* [ ] Каждый новый код покрыт тестом нужного слоя.
* [ ] Один PR содержит одну логически завершённую задачу.
* [ ] Документация/комментарии к публичным API обновлены.
* [ ] Временные решения помечены и описаны.
* [ ] Коммиты — только Conventional Commits (см. [`docs/git-workflow/commits.md`](docs/git-workflow/commits.md)).
* [ ] PHPUnit и Psalm выполнены успешно (или обоснованно пропущены по исключениям).
* [ ] Коммит и push выполнены по явному запросу пользователя, а не автоматически.
* [ ] Для завершённой задачи: `Статус` переведён в `done`, файл перенесён в `todo/done/`, поле `PR` заполнено.
* [ ] После переноса задачи обновлены ссылки в Epic и связанных задачах (новый путь `done/...`).
* [ ] Перед `merge`: задача переведена в `done`.
* [ ] После merge: `git status` выполнен, незакоммиченные файлы — согласованы с пользователем.
