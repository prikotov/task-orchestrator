# Архитектурное ревью: Agno Comparison Research

**Роль:** Архитектор Локи
**Дата:** 2026-04-22
**Объект:** `docs/research/agno-comparison.md`
**Задача:** Критический анализ исследования Agno — слепые зоны, слабые аргументы, риски заимствования, упущенные антипаттерны

---

## Резюме

Исследование добросовестно каталогизирует возможности Agno и формирует понятную таблицу «есть/нет/позже». Но как архитектурный документ, на основе которого будут приниматься решения о развитии системы, он содержит серьёзные пробелы. Основная проблема: **документ описывает «что у Agno есть», но не доказывает «почему это нам нужно» и «как это ляжет на нашу архитектуру».** Рекомендации не ранжированы по стоимости внедрения, а фундаментальная разница парадигм (Python SDK ↔ PHP CLI-orchestrator) системно недооценена.

Ниже — 23 находки, разбитые на 4 категории.

---

## 1. BLIND_SPOT — Слепые зоны

### BLIND_SPOT-01: Фундаментальная разница парадигм не проанализирована

**Суть:** Agno — Python SDK, работающий **in-process** с LLM API. task-orchestrator — PHP CLI-утилита, запускающая **внешние процессы** (pi, codex) и парсящая их вывод. Это не просто «разный подход» — это две принципиально разные модели исполнения.

**Что упущено:**
- Agno контролирует каждый LLM-вызов: он видит промпт, ответ, токены, latency — всё. Наш orchestrator видит только stdout внешнего CLI-процесса. Мы **не можем** реализовать половину фич Agno (compression, guardrails, memory, tool hooks) без изменения контракта с runner'ами.
- Многие фичи Agno (tool_hooks, guardrails, compression) работают именно потому, что LLM-вызовы **внутри** процесса. В нашей модели — runner — чёрный ящик, который получает промпт и возвращает текст. Мы не можем ставить middleware «вокруг tool calls», потому что мы не видим tool calls.
- Документ нигде не обсуждает этот ограничивающий фактор.

**Риск:** Читатель документа сделает вывод «нам нужно добавить X из Agno», не осознавая, что X требует фундаментальной перестройки контракта runner ↔ orchestrator.

---

### BLIND_SPOT-02: Стоимость внедрения не оценена ни для одной рекомендации

**Суть:** В таблице рекомендаций (раздел 5) 15 позиций со статусом `🟡`. Ни для одной не указана даже приблизительная оценка effort'а: это новый VO + 2 класса, или это 3-месячный проект с изменением ядра?

**Что нужно:**
- `Loop с end_condition` → вероятно, расширение `FixIterationGroupVo` + новый сервис. Низкая стоимость.
- `Conditional branching` → новая модель выполнения chain (сейчас — линейная/loop). Высокая стоимость, ломает ядро.
- `Parallel execution` → требуется параллельный процесс-менеджмент в PHP. Высокая стоимость + нетривиально в CLI.
- `HITL` → требует интерактивного взаимодействия с пользователем **во время** выполнения chain. Полная перестройка execution model.
- `Compression` → требует доступа к промежуточным LLM-вызовам внутри runner. **Невозможно** без изменения контракта runner.

**Риск:** Без оценки стоимости приоритизация бессмысленна. P2/P3 — это не приоритет, это откладывание решения.

---

### BLIND_SPOT-03: Отсутствие анализа «а нам это вообще нужно?»

**Суть:** Документ методично перечисляет, чего у нас нет и что есть у Agno. Но нигде не поставлен вопрос: **а какие реальные проблемы пользователей решает каждая фича?** Это каталог возможностей, а не анализ потребностей.

**Примеры:**
- `Memory system` (CRUD, semantic search). Для CLI-утилиты, запускающей цепочки агентов — зачем? Какой use case?
- `TeamMode: broadcast` — отправить задачу всем members одновременно. Когда нам это нужно?
- `Evaluation framework` (LLM-based pre/post checks). У нас уже есть quality gates (детерминированные). Зачем LLM-based проверки? Какой сценарий не покрывается shell-командами?
- `Nested workflows` (до 10 уровней). Это реально нужно? Или «было бы круто»?

**Риск:** Каталог фич превращается в roadmap без обоснования. Команда начнёт реализовывать «потому что у Agno есть», а не «потому что решает проблему».

---

### BLIND_SPOT-04: YAML vs Code — конфликт выразительности

**Суть:** Наши chains описываются в YAML. Agno описывает workflows **в Python-коде**. Это фундаментальная разница, которая определяет, что можно и что нельзя выразить.

**Что упущено:**
- `end_condition` как callable/lambda — не выражается в YAML. Только именованные условия или предопределённые стратегии.
- `Router` с `selector` как lambda — аналогично. YAML потребует декларативного DSL.
-CEL выражения — документ сам рекомендует «не брать CEL», но не объясняет, **чем** заменить callable для YAML.

**Риск:** Рекомендация «Loop с end_condition (callable)» по сути предлагает добавить в YAML мини-язык условий. Это может стать случайно сложным DSL, который никто не хочет поддерживать.

---

### BLIND_SPOT-05: Текущая модель fallback не проанализирована достаточно глубоко

**Суть:** Документ утверждает: «У нас уже есть fallback routing» и ставит ✅. Но наш fallback — это **static fallback** (заранее заданная команда для роли). Agno предлагает **error-specific dynamic fallback** (разные модели при разных ошибках). Это качественно разные вещи.

**Что упущено:**
- Наш `FallbackConfigVo` содержит только `command` (одна команда). Нет разделения по типу ошибки.
- Наш `ResolveChainRunnerService` пытается fallback при **любой** ошибке. Нет различения: rate limit → дешёвая модель, timeout → быстрый runner, context overflow → runner с другим model.
- Circuit Breaker и Fallback работают **параллельно**, но их взаимодействие не документировано. Что происходит: circuit breaker открылся → fallback пытается → fallback тоже упал → ?

---

### BLIND_SPOT-06: Отсутствие анализа устойчивости Agno как проекта

**Суть:** Agno (бывший Phi) прошёл ребрендинг. Версия 2.5.17 — молодая кодовая база. Документ не анализирует:
- Стабильность API (backward compatibility policy?)
- Сообщество и коммерческую поддержку
- Velocity изменений (сейчас активно развивается — API может ломаться)
- Готовность к production (1.7k+ GitHub stars — неплохо, но индикатор не definitive)

**Риск:** Заимствование паттернов из проекта с нестабильным API может привести к тому, что паттерны устареют или изменятся до нашей реализации.

---

### BLIND_SPOT-07: Parallel execution в PHP CLI — технически нетривиально

**Суть:** Рекомендация `Parallel execution в chains (P3)` упоминается как «востребованное расширение». Но параллельное выполнение внешних CLI-процессов в PHP — это не `Parallel` в Python (где есть GIL, но есть asyncio/multithreading/multiprocessing).

**Что нужно учесть:**
- PHP не имеет встроенного async runtime. `proc_open` + `stream_select` — вариант, но сложнее в поддержке.
- Parallel execution означает параллельный запуск **нескольких runner'ов** с各自的 prompt, output, error handling.
- Логирование (JSONL audit) должно корректно работать при параллельных процессах.
- Budget control должен учитывать параллельные вызовы (atomic budget checks).
- Session state (для resume) становится значительно сложнее.

**Риск:** Поставить `P3` на параллельное выполнение — это недооценить сложность. Реально это может оказаться сложнее, чем conditional branching.

## 2. WEAK_ARGUMENT — Слабые аргументы

### WEAK_ARGUMENT-01: «Loop с end_condition — прямое усиление fix_iterations»

**Цитата:** «Прямое усиление нашего fix_iterations: сейчас только max_iterations, а с end_condition — детерминированная проверка завершения.»

**Почему слабый аргумент:**
1. Наш `FixIterationGroupVo` уже поддерживает именованные группы шагов и `maxIterations`. Условие завершения — это **качественное изменение модели**, а не «усиление». Сейчас цикл выполняется N раз. С end_condition — цикл выполняется «пока условие истинно». Это разные модели управления потоком.
2. Непонятно, **кто** вычисляет `end_condition`. В Agno — Python callable. У нас — YAML. Предложение «достаточно callable-условий» противоречит YAML-first подходу.
3. Не проанализировано, как `end_condition` взаимодействует с quality gates. Сейчас quality gate после шага — это implicit «продолжать или нет». С end_condition появятся два механизма остановки.

---

### WEAK_ARGUMENT-02: «Error-specific fallback — дополнение к circuit breaker»

**Цитата:** «Наш CircuitBreakerAgentRunner защищает от cascade failures, но не переключает на альтернативный runner. Agno показывает, как добавить error-specific fallback.»

**Почему слабый аргумент:**
1. Circuit Breaker и error-specific fallback решают **разные** проблемы, но документ это не объясняет. CB — «не ходи к мёртвому сервису». Error-specific fallback — «ходи к другому, если этот вернул конкретную ошибку».
2. Наш runner — CLI-процесс. Мы не всегда можем определить **тип** ошибки (rate limit vs timeout vs context overflow). Pi CLI может вернуть exit code 1 в любом случае. Различение типов ошибок требует контракта с runner'ом, которого у нас нет.
3. «Дополнение к circuit breaker» звучит как маленькое изменение. На практике — это новый layer маршрутизации с пониманием error semantics.

---

### WEAK_ARGUMENT-03: «Tool hooks — аналог нашего decorator pattern, но на уровне tool calls»

**Цитата:** «Для chain executor tool_hooks может заменить декораторы для per-step middleware.»

**Почему слабый аргумент:**
1. Это **не аналог**. Наш decorator pattern работает на уровне **runner'ов** (AgentRunnerInterface). Tool hooks в Agno работают на уровне **LLM tool calls внутри агента**. Это разные уровни абстракции.
2. Мы **не видим** tool calls. Runner — внешний CLI-процесс. Мы не можем поставить hook вокруг tool call, который происходит внутри pi/codex.
3. «Заменить декораторы» — это предложение убрать работающий паттерн и заменить его на неподходящий аналог.

---

### WEAK_ARGUMENT-04: «Compression — менее радикально, чем auto-summarization»

**Цитата:** «Сжатие tool results — самый простой первый шаг: не сжимать весь контекст, а только output tool calls.»

**Почему слабый аргумент:**
1. CompressionManager Agno сжимает **LLM tool call results** — это результаты функций, которые вызывает LLM внутри сессии. У нас нет LLM tool calls в этом смысле. У нас есть «output агента» — текст, который runner возвращает.
2. Сжатие output агента через LLM — это по сути **перегенерация** результата. Стоимость: ещё один LLM-вызов. Ради сжатия контекла?
3. Проблема растущего контекста в dynamic loops — реальная. Но решение «сжать output через LLM» — самое дорогое из возможных. Дешевле: truncate, sliding window, structured output.

---

### WEAK_ARGUMENT-05: «Guardrails — pre-flight checks перед выполнением шага»

**Цитата:** «Для production: проверка входных данных (нет ли PII, не содержит ли prompt injection) перед отправкой в runner.»

**Почему слабый аргумент:**
1. PII detection и prompt injection detection — это LLM-powered checks. То есть мы предлагаем **сделать LLM-вызов** для проверки промпта перед тем, как **сделать LLM-вызов** для выполнения. Это удвоение стоимости и latency.
2. В нашей модели orchestrator отправляет **role prompt + context** внешнему CLI-процессу. Промпт формируется из наших же `.md` файлов. Мы **контролируем** промпт. Prompt injection — риск на уровне пользовательского ввода, а не на уровне наших промптов.
3. PIIDetectionGuardrail в Agno проверяет **пользовательский ввод**, поступающий в LLM. У нас пользовательский ввод — это задача (task description). Это валидация Presentation-слоя, а не Domain.

---

### WEAK_ARGUMENT-06: «Typed I/O per step — валидация входных/выходных данных»

**Эта рекомендация даже не обоснована в документе.** Она просто появляется в таблице рекомендаций (раздел 5) без описания в разделе 3. Typed I/O — это хорошая практика, но:
1. Agno не имеет typed I/O per step в том смысле, в котором это описано. Steps в Agno — это обёртки вокруг agents/teams/executors. I/O — это chat messages.
2. У нас каждый шаг — это вызов runner'а. Вход: промпт + контекст. Выход: текст + токены + cost. Typed I/O означал бы **схему** для каждого шага. Но LLM-вывод — это по определению неструктурированный текст. Схема для output — это по сути quality gate.

## 3. ADOPTION_RISK — Риски заимствования

### ADOPTION_RISK-01: Workflow Engine сломает линейную модель Chain

**Риск:** Agno Workflow (Step/Loop/Parallel/Router/Condition) — это полноценный workflow engine. Наша цепочка — **линейная** последовательность шагов с optional fix_iteration loop. Внедрение Router и Condition превращает chain из списка в **граф выполнения**.

**Архитектурные последствия:**
- `ChainDefinitionVo` и `ChainStepVo` потребуют переработки. Сейчас step = элемент списка. С Router/Condition — step = node в графе.
- `ExecuteStaticChainService` / `RunDynamicLoopService` — одна модель выполнения. С графом — нужна абстракция execution engine.
- YAML-схема для цепочек станет значительно сложнее (conditions, routers, parallels — всё в YAML).
- Session state (resume) для графа выполнения — на порядок сложнее, чем для линейного списка.

**Вердикт:** Это не «P2» — это архитектурное решение, которое нужно принимать как ADR с полным обоснованием.

---

### ADOPTION_RISK-02: HITL требует изменения execution model

**Риск:** Human-in-the-Loop означает, что выполнение цепочки **останавливается** и ждёт ввода пользователя. Наша модель: CLI-команда запускается, выполняет chain от начала до конца, возвращает результат.

**Архитектурные последствия:**
- CLI-процесс должен уметь «спать» и «просыпаться» при вводе. Это интерактивный режим.
- Session persistence становится **обязательной** (сейчас опциональна для resume). Без persistence HITL невозможен — процесс может упасть, пока ждёт ввода.
- Таймауты HITL (`hitl_timeout`, `on_timeout`) — ещё один layer управления процессом.
- Наш `OrchestrateChainCommandHandler` сейчас синхронный: получил команду → выполнил → вернул результат. HITL требует асинхронной модели.

**Вердикт:** Реализация HITL в PHP CLI — это не фича, это смена парадигмы выполнения. Оценка: минимум 2-3 недели на ядро.

---

### ADOPTION_RISK-03: Team Mode конфликтует с нашей моделью runner

**Риск:** Agno Team — это несколько **LLM-агентов** (in-process), координируемых одним supervisor-агентом. У нас «агент» = внешний CLI-процесс. «Команда» = несколько CLI-процессов.

**Архитектурные последствия:**
- `coordinate` mode: supervisor выбирает member и формулирует задачу. У нас это dynamic chain с facilitator — уже реализовано. Но facilitator — тоже CLI-процесс. Agno supervisor — Python-объект с прямым доступом к LLM API.
- `route` mode: routing к конкретному агенту. У нас это `runner` parameter в `ChainStepVo`. Реализовано на другом уровне.
- `broadcast` mode: параллельная отправка всем. См. ADOPTION_RISK-01 (parallel execution) + проблема координации результатов от нескольких CLI-процессов.
- `tasks` mode: autonomous decomposition. Это уже близко к нашему dynamic loop, но Agno делает это через in-process LLM calls, а не через внешние CLI-процессы.

**Вердикт:** Заимствование Team Modes без учёта разницы «in-process LLM» vs «external CLI process» приведёт к «дизайну, который выглядит как Agno, но работает по-другому».

---

### ADOPTION_RISK-04: Memory и Session Persistence — чужая парадигма

**Риск:** Документ упоминает «Session persistence (12+ адаптеров)» и «Memory system» как интересные. Но:
1. Наша сессия — это JSONL-файл в `var/agent/chains/`. Для resume этого достаточно. Встроенная session persistence у нас **уже есть** (ChainSessionWriter/Reader).
2. Memory system Agno — это semantic search по предыдущим взаимодействиям пользователя с агентом. У нас нет «пользовательских сессий» в этом смысле — мы CLI-утилита.
3. Добавление 12+ адаптеров хранения — это overengineering для CLI. PostgreSQL для CLI-утилиты? SQLite — может быть, но зачем, если JSONL работает?

**Вердикт:** Сравнение «у нас нет session persistence» (в таблице стоит ❌) **фактически неверно**. У нас есть session persistence через JSONL и ChainSessionWriter/Reader. Другое дело, что это не «pluggable storage с 12 адаптерами» — но это и не нужно.

---

### ADOPTION_RISK-05: «У нас лучше» — ложное чувство превосходства

**Риск:** Документ помечает 7 позиций как «✅ У нас лучше» (retry, circuit breaker, quality gates, budget, roles, DDD, decorator). Это создаёт опасную иллюзию: «мы впереди по основам, берём только расширения».

**Реальность:**
- **Retry**: у нас exponential backoff — да, лучше базово. Но Agno retry — на уровне step, наш — на уровне runner call. Разные уровни.
- **Quality Gates**: наши — shell-команды. Agno guardrails + evals — LLM-based проверки. Это не «лучше/хуже» — это **ортогональные** подходы.
- **DDD**: «у нас лучше» — но Agno не претендует на DDD. Сравнивать архитектурную дисциплину PHP-бандла с Python SDK-framework — сравнивать тёплое с мягким.
- **Roles**: наши `.md` файлы — это промпты. Agno `instructions` — это тоже промпты. «У нас лучше» — потому что 18 файлов? Это не архитектурное преимущество.

**Вердикт:** «У нас лучше» должно быть обосновано конкретными метриками или сценариями, а не вкусом.

---

## 4. ANTIPATTERN — Упущенные антипаттерны Agno

### ANTIPATTERN-01: Agent — God Object (1729 строк)

**Суть:** `agent/agent.py` — один dataclass на 1729 строк. Содержит: generate, run, tools, memory, hooks, guardrails, evals, reasoning, compression, session management. Это классический God Object.

**Почему документ это не отмечает:** Документ перечисляет все фичи Agent как «ключевые характеристики» без критики реализации.

**Риск заимствования:** Если мы начнём добавлять hooks, guardrails, evals в шаг цепочки — `ChainStepVo` (сейчас ~120 строк) может вырасти до аналогичного God Object. Нужно продумать extension mechanism (composition, не accretion).

---

### ANTIPATTERN-02: Team — Supervisor Bottleneck

**Суть:** В режимах `coordinate` и `tasks` supervisor-агент единолично принимает решения о маршрутизации. При 5+ members с complex tasks — это bottleneck. Supervisor сам делает LLM-вызов для каждого routing decision.

**Почему документ это не отмечает:** `coordinate` описан как «supervisor pattern» без анализа ограничений.

**Риск заимствования:** Наш dynamic chain с facilitator — та же модель. Если цепочка растёт (10+ ролей), facilitator станет bottleneck. Документ не анализирует масштабируемость.

---

### ANTIPATTERN-03: Fallback без Circuit Breaker

**Суть:** Agno `FallbackConfig` переключает на fallback при ошибке, но **не имеет circuit breaker**. Это означает: при persistent failure primary model → fallback model → fallback model тоже → ... → нет защиты от cascade.

**Почему документ это не отмечает:** Документ противопоставляет «у нас CB, у них error-specific fallback» как два разных подхода, не замечая, что Agno **не имеет** CB. Это не «у них богаче», а «у них другой trade-off».

**Риск заимствования:** Добавить error-specific fallback без CB — потерять protection. Добавить и то, и другое — сложная конфигурация. Документ не предлагает, как их совместить.

---

### ANTIPATTERN-04: Mutable State в Dataclasses

**Суть:** Agno активно использует Python dataclasses с mutable state (session state, memory, running counters). Agent и Team хранят runtime state в своих полях. Это затрудняет testing, debugging, и параллельное использование.

**Почему это важно для нас:** Наш подход — immutable VO (`readonly class`) и in-memory entities. Это осознанный архитектурный выбор. Если мы начнём добавлять stateful features (memory, session) «по мотивам Agno», есть риск нарушить immutability without noticing.

---

### ANTIPATTERN-05: Tight Coupling между Workflow Components

**Суть:** Agno Workflow шаги (Step, Loop, Parallel, Router, Condition) тесно связаны: Router содержит choices (список Step), Loop содержит steps (список Step), Parallel содержит steps. Нет общего абстрактного «workflow node» — каждый тип шага — отдельный dataclass с дублированной логикой (HITL, retries, session persistence).

**Почему документ это не отмечает:** Документ описывает 6 «строительных блоков» как элегантную систему, не замечая дублирования.

**Риск заимствования:** Если мы пойдём по пути «Loop, Router, Condition — отдельные классы», получим 6+ сущностей с дублированной логикой. Лучше: единая абстракция `ChainNode` с разными стратегиями выполнения.

---

### ANTIPATTERN-06: «Всё включено» — отсутствие границ

**Суть:** Agno стремится быть всем для всех: LLM integration, RAG, vector DB, memory, guardrails, scheduling, UI, monitoring. Это приводит к огромной кодовой базе с большим числом зависимостей и неясными границами ответственности.

**Почему это важно для нас:** Документ рекомендует «не брать» 5 фич, но **все остальные** помечены как P2-P3. По сути предлагается заимствовать 15+ фич из фреймворка, который сам страдает от lack of focus. Наша сила — **узкий фокус**: orchestration of external CLI agents. Расширение scope «поAgno» размоет этот фокус.

### Что стоит сделать с документом

### Что стоит взять из Agno реально

Из всего каталога я вижу **три** обоснованных заимствования:

1. **Loop с end_condition** (стратегия, не callable) — расширить `FixIterationGroupVo` предопределёнными стратегиями завершения: `all_quality_gates_pass`, `no_errors`, `budget_remaining`. Это YAML-friendly, не ломает архитектуру, решает реальную проблему (бессмысленные итерации после успеха). **Effort: малый.**

2. **Error-specific fallback types** — расширить `FallbackConfigVo` поддержкой типов ошибок, которые мы **можем** различать (timeout, non-zero exit code, empty output). Не требуются типы из Agno (rate_limit, context_overflow), которые мы не можем определить. **Effort: малый.**

3. **Nested chains** (chain как шаг другой chain) — это естественно ложится в нашу модель: `ChainStepVo` получает тип `chain` и ссылку на другую chain по имени. Session state и logging уже есть. **Effort: средний.**

Остальное — либо невозможно без изменения контракта runner, либо не решает реальных проблем, либо требует смены парадигмы выполнения. Не стоит того.
