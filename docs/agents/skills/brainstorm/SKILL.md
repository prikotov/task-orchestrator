# Brainstorm

Автономная brainstorm-сессия с конструктивным конфликтом. Фасилитатор разжигает спор, участники защищают позиции и оспаривают чужие, финал — протокол собрания с решениями, владельцами и планом действий.

## Когда использовать

- Нужно принять архитектурное или продуктовое решение с несколькими альтернативами
- Есть спорный вопрос, где важно услышать разные точки зрения
- Требуется протокол обсуждения с конкретными решениями и планом действий
- Планирование эпика или релиза, где нужны аргументы «за» и «против»

## Как использовать

### **Шаг 1:** Формулировка темы

Сформулируй тему brainstorm — один-два предложения, описывающие проблему или вопрос для обсуждения.

**Требования к теме:**
- Конкретная, а не абстрактная (плохо: «архитектура», хорошо: «какие модули выделить из AgentRunner»)
- Содержит контекст, чтобы участники могли сразу включиться
- Формулируется на языке, понятном всем участникам

### **Шаг 2:** Выбор участников

По умолчанию brainstorm использует участников из цепочки `brainstorm` в `config/chains.yaml`.

Переопределение через опции CLI:

| Опция              | Описание                                     | По умолчанию (chains.yaml)                                              |
|--------------------|----------------------------------------------|-------------------------------------------------------------------------|
| `--chain`          | Имя цепочки                                  | `brainstorm`                                                            |
| `--facilitator`    | Роль фасилитатора                            | `team_lead_alex`                                                        |
| `--participants`   | Участники через запятую                      | `system_architect_gandalf,system_architect_loki,backend_developer_tony` |
| `--max-rounds`     | Макс. раундов обсуждения                     | `20`                                                                    |
| `--timeout`        | Таймаут на шаг, секунды                      | `600`                                                                   |

**Подбор участников:**

| Цель brainstorm          | Рекомендуемые участники                                                         |
|--------------------------|---------------------------------------------------------------------------------|
| Архитектурное решение    | `system_architect_gandalf`, `system_architect_loki`, `backend_developer_tony`   |
| Анализ требований        | `system_analyst_sherlock`, `system_architect_gandalf`, `system_architect_loki`  |
| Планирование реализации  | `system_architect_gandalf`, `backend_developer_levsha`, `backend_developer_tony`|

**Здоровый конфликт:** Гэндальф (системный стратег) + Локи (скептик, альтернативы) = конструктивное столкновение. Рекомендуется включать обоих в архитектурные brainstorm.

### **Шаг 3:** Запуск brainstorm

```bash
    php bin/console app:agent:orchestrate "<тема>" [опции]
```

Опции:

| Опция              | Сокращение | Описание                                     | Значения              | По умолчанию  |
|--------------------|------------|----------------------------------------------|-----------------------|---------------|
| `--chain`          | `-c`       | Имя цепочки                                  | имя цепочки           | `brainstorm`  |
| `--facilitator`    |            | Роль фасилитатора                            | имя роли              | из chains.yaml|
| `--participants`   |            | Участники через запятую                      | имена ролей           | из chains.yaml|
| `--max-rounds`     |            | Макс. раундов (dynamic)                      | число                 | из chains.yaml|
| `--timeout`        | `-t`       | Таймаут на шаг, секунды                      | число                 | из chains.yaml|
| `--topic`          |            | Тема (если отличается от аргумента task)     | текст                 | = task        |
| `--runner`         | `-r`       | Движок AI                                    | `pi`                  | `pi`          |
| `--model`          | `-m`       | Модель LLM                                   | имя модели           | по умолчанию  |
| `--dry-run`        |            | Показать план без запуска                    | —                     | выкл          |
| `--report-format`  |            | Формат отчёта                                | `text`, `json`, `none`| `text`        |
| `--report-file`    |            | Путь к файлу отчёта                          | путь                  | stdout        |
| `--no-audit-log`   |            | Отключить audit-логирование                  | —                     | выкл          |
| `--no-context-files`|           | Отключить загрузку AGENTS.md/CLAUDE.md       | —                     | выкл          |
| `--resume`         |            | Путь к директории сессии для resume          | путь                  | —             |
| `--working-dir`    | `-d`       | Рабочая директория                           | путь                  | корень проекта|

Примеры:

```bash
    # Стандартный brainstorm с параметрами из chains.yaml
    php bin/console app:agent:orchestrate "Какие модули выделить из AgentRunner"

    # Переопределение участников и лимита раундов
    php bin/console app:agent:orchestrate "Стратегия тестирования" \
        --participants=system_analyst_sherlock,system_architect_gandalf,system_architect_loki \
        --max-rounds=15

    # Другой фасилитатор + отчёт в файл
    php bin/console app:agent:orchestrate "Кэширование в оркестраторе" \
        --facilitator=team_lead_alex \
        --participants=backend_developer_levsha,system_architect_gandalf \
        --report-format=text \
        --report-file=docs/brainstorm/cache-strategy.md

    # Dry run — посмотреть план без запуска
    php bin/console app:agent:orchestrate "Паттерн retry" --dry-run

    # Resume прерванной сессии
    php bin/console app:agent:orchestrate "Продолжение brainstorm" \
        --resume=var/sessions/brainstorm/2026-04-23_03-02-36
```

### **Шаг 4:** Анализ результата

После завершения brainstorm:

1. Проверь код возврата: `0` — сессия завершена с синтезом (протоколом), `1` — прервана (таймаут, бюджет, ошибка).
2. Прочитай протокол из отчёта (stdout или файл из `--report-file`).
3. При необходимости — изучи audit-лог в директории сессии (`var/sessions/<chain>/<timestamp>/`).

**Структура протокола:**
- Участники и повестка
- Ключевые разногласия и разрешённые конфликты
- Принятые решения с обоснованием
- План действий (Action Items) с ответственными

### **Шаг 5:** Работа с результатом

На основе протокола brainstorm:

1. Зафиксируй принятые решения в соответствующих документах (архитектура, задачи, epic).
2. Создай задачи на Action Items из плана действий — каждую с ответственным и критерием готовности.
3. Если brainstorm проводился в рамках эпика — обнови план эпика с учётом решений.

## Результат

**Основной:** Протокол собрания (stdout или файл из `--report-file`).

Поля протокола:

| Поле                     | Описание                                                        |
|--------------------------|-----------------------------------------------------------------|
| Участники                | Кто участвовал и в какой роли                                   |
| Повестка                 | Формулировка темы обсуждения                                    |
| Ключевые разногласия     | Где участники не пришли к согласию                              |
| Разрешённые конфликты    | Где спор привёл к новому пониманию                              |
| Открытые вопросы         | Неразрешённые разногласия                                       |
| Принятые решения         | Решение + обоснование + альтернатива                            |
| План действий            | Задача + ответственный + приоритет + критерий готовности        |

**Побочные:**

| Ресурс                                      | Описание                                |
|---------------------------------------------|------------------------------------------|
| `var/sessions/<chain>/<timestamp>/`         | Директория сессии с audit-логом и шагами |
| `var/sessions/<chain>/<timestamp>/audit.jsonl` | Полный аудит всех вызовов агентов      |
