# Запрос на слияние (Pull Request)

**Pull Request (PR)** — предложенный набор изменений из рабочей ветки в целевую ветку (`master` или активную `release/x.y`), проходящий проверки и `Code Review`.

## Границы ответственности

- Документ описывает подготовку PR, выбор base branch и требования к содержимому PR.
- Правила веток описаны отдельно: [Ветки (Branches)](branches.md).
- Правила коммитов описаны отдельно: [Коммиты (Commits)](commits.md).
- Правила релизов описаны отдельно: [Релизы (Release)](release.md).
- Как проводить ревью изменений описано отдельно: [Ревью кода (Code Review)](code-review.md).

## Правила

- Запрещены прямые правки в `master`.
- Одна задача — один PR, либо одна подзадача, если задача большая.
- Не смешивай изменения из разных задач.
- Держи PR минимальным по объёму.
- Не добавляй в PR изменения "заодно".
- Не тащи в PR форматирование без смысла и правки из других задач.
- Убирай из PR debug-код и “строительный мусор”.
- PR должен быть самодостаточным после merge в свою целевую ветку.
- Не делай зависимые PR, которые по отдельности не работают.
- Не накапливай изменения в долгоживущей ветке: всё, что можно безопасно смержить в целевую ветку, оформляй отдельным PR.
- Не проси пользователя создать PR и не ссылайся на “нет прав”.
- Не смешивай `move/rename` и последующий `refactor/behavior change`:
  - перенос, переименование и массовые механические правки — отдельный PR;
  - рефакторинг и изменение поведения — отдельный PR.
- Ориентир размера PR: около 600 строк `diff` (`added + deleted`).
- Если PR больше 600 строк — разбей его или объясни причину в описании и добавь навигацию по ревью.
- PR заглушек или архитектурных изменений может быть больше при формальных правках; PR с новой логикой — наоборот, держи компактным.
- Значимые изменения сопровождай тестами соответствующего уровня.
- Любое временное решение или архитектурный компромисс помечай в коде и описывай в PR.

## Выбор base branch

- `task/*` для обычной разработки → `master`.
- Stabilization fixes для текущего релиза → активная `release/x.y`.
- `hotfix/*` → активная `release/x.y`, если release line ещё жива; дополнительно обязателен merge-back в `master`.
- Если активная `release/x.y` уже закрыта, hotfix оформляется как patch line от production tag, а merge-back в `master` выполняется отдельным PR сразу после выпуска patch release.

## Создание PR

- Определи, это `docs-only` правка или нет.
- Если это не `docs-only` — **обязательно** запусти `make check`.
- `make check` должен быть зелёный.
- Если это `docs-only` — укажи в PR, что `make check` пропущен по исключению.
- Сделай self-review: [Ревью кода (Code Review)](code-review.md).
- После создания PR синхронизируй задачу в `todo/`: заполни поле `pr`, переведи `Статус` в `review`.
- Используй CLI-инструменты: `git`, `gh`.
- Сделай push: `git push -u origin HEAD`.
- Создай PR через `gh pr create --head $(git branch --show-current)`.
- Тело PR передавай через `gh pr create --body-file`.
- Предпочитай `--body-file -` (stdin) или файл в `tmp/`.
- После создания PR обнови поле `PR` в задаче (`todo/...`) ссылкой или номером созданного PR.

## Оформление PR

- Заголовок PR: английский, кратко отражает суть.
- Для release и hotfix PR явно укажи:
  - release line (`release/x.y`);
  - production tag, который исправляем или выпускаем;
  - нужен ли merge-back в `master`;
  - какие изменения сознательно **не** входят в этот релиз.
- Описание PR включает:
  - если задачи в `todo/` ещё нет — оформи её по правилам: [todo/AGENTS.md](../../todo/AGENTS.md);
  - постановку задачи из prompt;
  - постановку задачи из `todo/<ID>.todo.md` (если есть);
  - цель и причину изменений;
  - основные модули и файлы;
  - особые случаи и breaking changes;
  - ссылку на задачу или issue;
  - проверки и результат с кратким саммари `stdout` или ссылкой на CI job.
- Метка AI-агента **обязательна** (label в GitHub).
- Выбирай метку AI-агента по использованному инструменту.
  - `gemini-cli` — Gemini CLI
  - `codex-cli` — Codex из локального CLI (для Codex: используй по умолчанию, если сессия запущена пользователем из CLI)
  - `codex` — Codex из cloud-окружения (для Codex: используй только если работа явно идёт не из локального CLI)
  - `opencode` — OpenCode
  - `roocode` — Roo Code
  - `kilocode` — Kilo Code
  - `pi` — Pi Coding Agent
  - Если нужной метки нет — добавь её в репозиторий.
- Не добавляй несколько “агентских” меток в один PR.
- Укажи метку при создании: `gh pr create --label "<agent-label>"`.
- Или добавь метку после: `gh pr edit <PR_NUMBER> --add-label "<agent-label>"`.

## Перед merge

- **До** вызова `gh pr merge` переведи задачу в `done` прямо в ветке PR:
  - обнови `status: done` в файле задачи;
  - перенеси файл из `todo/` в `todo/done/`;
  - обнови ссылки в Epic и зависимых задачах;
  - закоммить и запушь изменения в ту же ветку.
- Это гарантирует, что обновление задачи попадёт в целевую ветку только через PR.

## Выполнение merge

- После approval пользователя заверши PR через GitHub (`gh pr merge`).
- Запрещено выполнять локальный merge PR-ветки в целевую ветку.
- После merge hotfix в `release/x.y` не забудь сделать merge-back в `master`.

## После merge

- После merge PR в `master` переключись на `master` и обнови его: `git pull origin master`.
- После merge PR в `release/x.y` переключись на эту release line только если продолжается стабилизация; иначе вернись в `master`.
- Удали рабочую ветку локально и в `origin`.
- Проверь, что рабочее дерево чистое: `git status`.
- Предложи пользователю следующую задачу.
